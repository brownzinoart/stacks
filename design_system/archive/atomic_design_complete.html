Content structure patterns that articulate layout and data requirements

What are Templates?

Templates are page-level wireframes that show content structure, not final design. They define what content is needed, how much content to support, and how components are arranged.

üìù Brad Frost's Key Principle:

"You can create good experiences without knowing the content. What you can't do is create good experiences without knowing your content structure."

Dashboard Template

Main user dashboard with reading progress, stats, and current book

MOBILE TOP BAR
  • Component: mobile-top-bar (organism)
  • Required: Always present
  • Contains: Brand text + Avatar
HERO SECTION
[TITLE: 30-50 characters]
[DESCRIPTION: 80-150 characters]
[BADGE: 1 status badge]
  • Component: mobile-hero (organism)
  • Background: Gradient (dynamic based on status)
  • Height: 180-220px
STATS GRID
[STAT_CARDS: 2-4 cards, 2-column grid]
  • Each Card: [NUMBER: 1-4 digits] + [LABEL: 10-20 chars]
  • Background: Gradient per card
  • Required Cards: Minimum 2
PROGRESS SECTION
[SECTION_TITLE: 20-40 characters]
[PROGRESS_CARD: Contains book progress]
  • Progress Data: [CURRENT_PAGE] / [TOTAL_PAGES]
  • Progress Bar: 0-100% fill
  • Label: "pages read"
BOTTOM NAVIGATION
  • Component: bottom-nav (organism)
  • Items: Exactly 5 tabs
  • Active State: 1 tab highlighted
  • Position: Fixed to bottom
// Template Structure (Pseudocode) template Dashboard { components: [ { type: "mobile-top-bar", required: true }, { type: "mobile-hero", required: true, content: { title: { maxLength: 50 }, description: { maxLength: 150 }, badge: { count: 1 } } }, { type: "stats-grid", required: true, range: { min: 2, max: 4 } }, { type: "bottom-nav", required: true } ] }

List View Template

Vertical list of items (library, book queue, etc.)

[PAGE_TITLE: 20-30 characters]
[ACTION_BUTTON: 1 FAB or icon button]

[COLLECTION_NAME: 30-50 characters]
[ITEM_COUNT: Number + label, e.g. "3 Borrowed"]
[DESCRIPTION: 80-120 characters]
LIST ITEMS
[ITEM_LIST: Supports 0-100 items]
  • Each Item: mobile-card (organism)
  • Image: [IMAGE: 1:1 ratio, 80-120px, optional]
  • Title: [TITLE: 40-80 characters]
  • Metadata: [AUTHOR/INFO: 30-60 characters]
  • Progress: [PROGRESS_BAR: 0-100%, optional]
  • Action: [CTA_BUTTON: 1 button]
EMPTY STATE
  • Trigger: When item_count = 0
  • Message: [EMPTY_MESSAGE: 60-120 characters]
  • Icon: [EMOJI or illustration]
  • Action: [CTA: "Add First Item" button]
BOTTOM NAVIGATION
  • Active Tab: Matches current view

Development workflow for building the design system (Frontend & Backend)

Development Workflow

Based on Brad Frost's Chapter 4, this workflow emphasizes browser-based iterative development where frontend and backend developers work together from day one, building components in atomic order.

Frontend Implementation

Building UI components with HTML, CSS, and JavaScript

1
Setup Development Environment
  • Initialize Git repository
  • Set up build tools (Vite, Webpack, or similar)
  • Configure hot reloading for instant feedback
  • Set up CSS preprocessing if needed (Sass, PostCSS)
// Package.json scripts { "scripts": { "dev": "vite", "build": "vite build", "preview": "vite preview" } }
2
Build Atoms First
  • Start with foundational CSS (colors, typography, spacing)
  • Create button styles with all variants
  • Build input field styles
  • Design badge components
  • Test in browser immediately - don't wait
// atoms/button.css .btn { padding: 18px 36px; border: 4px solid #000; border-radius: 12px; font-weight: 900; box-shadow: 4px 4px 0px #000; } .btn-primary { background: #667eea; } .btn-secondary { background: #ffe66d; }
3
Compose Molecules
  • Combine atoms into simple functional units
  • Test molecule behavior in isolation
  • Ensure single responsibility principle
  • Make molecules reusable across templates
// molecules/search-bar.html <div class="search-bar"> <input type="text" class="input" placeholder="Search..."> </div> // CSS adds the search icon via ::before
4
Build Organisms
  • Assemble organisms from molecules and atoms
  • Create distinct interface sections (header, card, form)
  • Test organisms with varying content lengths
  • Ensure responsive behavior across breakpoints
// organisms/mobile-card.html <div class="mobile-card"> <!-- Molecule: Card Header --> <div class="mobile-card-header"> <h3>{{title}}</h3> <span class="badge">{{badge}}</span> </div> <!-- Atom: Body text --> <p>{{description}}</p> <!-- Atom: Button --> <button class="btn btn-primary">{{ctaText}}</button> </div>
5
Create Templates
  • Stitch organisms together into page layouts
  • Define content structure and constraints
  • Use placeholder content to show structure
  • Document dynamic content ranges
// templates/dashboard.html <div class="page-dashboard"> {{> organisms/mobile-top-bar}} {{> organisms/mobile-hero}} {{> organisms/stats-grid}} {{> organisms/bottom-nav}} </div> // Using template partials/includes for reusability
6
Implement Pages
  • Populate templates with real content
  • Create multiple page instances showing variations
  • Test edge cases (empty states, long text, missing images)
  • Validate performance across devices

Backend Implementation

API structure and data modeling for atomic components

1
Component-Driven API Design
  • Structure API responses to match component needs
  • Return data in the shape components expect
  • Minimize frontend data transformation
// API Response for Dashboard Template { "hero": { "title": "CATCH UP!", "description": "You're behind schedule. Read 20 pages today!", "badge": { "text": "READING PLAN", "variant": "accent" } }, "stats": [ { "value": 14, "label": "Days Left", "gradient": "coral" }, { "value": 20, "label": "Pages/Day", "gradient": "yellow" } ], "progress": { "current": 120, "total": 400, "percentage": 30 } }
2
Respect Content Constraints
  • Enforce character limits defined in templates
  • Validate data against template requirements
  • Return appropriate defaults for optional fields
  • Handle empty states gracefully
// Backend Validation (Python example) class HeroContent: title: str = Field(max_length=50) description: str = Field(max_length=150) badge: Optional[Badge] = None def validate_length(self): if len(self.title) > 50: raise ValueError("Title exceeds template limit")
3
Component-Level Endpoints
  • Create endpoints that return organism-level data
  • Allow frontend to request specific components
  • Support partial page updates
// RESTful API Structure GET /api/dashboard // Full page GET /api/dashboard/hero // Just hero organism GET /api/dashboard/stats // Just stats organism GET /api/library/items // List organism GET /api/library/items/:id // Single item detail
4
Database Schema Alignment
  • Model database to support template structures
  • Store gradient preferences, badge states
  • Track content metadata (length, type)
// Database Schema (SQL example) CREATE TABLE books ( id UUID PRIMARY KEY, title VARCHAR(80), -- Matches template limit author VARCHAR(60), -- Matches template limit description TEXT, cover_image_url TEXT, pages_total INTEGER, pages_read INTEGER DEFAULT 0, status VARCHAR(20) -- 'reading', 'queue', 'finished' ); CREATE TABLE reading_stats ( user_id UUID, stat_type VARCHAR(30), -- 'days_left', 'pages_per_day' value INTEGER, gradient_color VARCHAR(20) -- 'coral', 'yellow', 'purple' );

Recommended Build Order

Following atomic principles, build from smallest to largest

WEEK 1: ATOMS

  • ‚úì Set up design tokens (colors, spacing, typography)
  • ‚úì Build button components (all variants)
  • ‚úì Build input components (text, email, textarea)
  • ‚úì Build badge components
  • ‚úì Test all atoms in Storybook or similar

WEEK 2: MOLECULES

  • ‚úì Form fields (label + input)
  • ‚úì Search bars (icon + input)
  • ‚úì Button groups
  • ‚úì Card headers (title + badge)
  • ‚úì Progress bars
  • ‚úì Bottom nav items

WEEK 3: ORGANISMS

  • ‚úì Mobile top bar
  • ‚úì Bottom navigation
  • ‚úì Hero sections (desktop + mobile)
  • ‚úì Content cards
  • ‚úì Forms
  • ‚úì Stats grid

WEEK 4: TEMPLATES & PAGES

  • ‚úì Dashboard template
  • ‚úì List view template
  • ‚úì Detail view template
  • ‚úì Wire up to backend APIs
  • ‚úì Test with real content variations
  • ‚úì Handle empty states, errors

üí° Brad Frost's Key Principle:

"As each pattern becomes more fully baked, any template that includes the pattern will become more fully baked as well." Build incrementally, test continuously.
